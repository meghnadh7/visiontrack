VisionTrack – Technical Report

Meghnadh Bijnapally meghnadh007@gmail.com April 2026

1. Project Overview

VisionTrack is a unified web dashboard I built to bridge two separate tools I was already using in my workflow — Roboflow for managing computer vision datasets and Redmine for tracking project issues. The idea was pretty simple: instead of switching between two browser tabs constantly, have one interface that lets me view datasets, run inference, and see which issues are open, all in one place.

The app ended up with five main areas: a dashboard overview, Roboflow project and image management, an inference runner, and Redmine project/issue tracking with a members panel. There's also a GitHub Actions CI pipeline and Docker setup for both services.

2. Architecture

I went with a Backend-for-Frontend (BFF) pattern. The React app never talks to Roboflow or Redmine directly — everything goes through an Express proxy that handles auth headers and any response normalisation. This kept API keys off the client entirely.

Browser (React + Vite) │ │ HTTP /api/* ▼ Express BFF (Node 20, TypeScript) ├── /api/roboflow/* ──────▶ api.roboflow.com (API key in query param) └── /api/redmine/* ──────▶ Redmine instance (X-Redmine-API-Key header) Redmine runs in a Docker container on port 8080 (SQLite storage).

One thing I didn't anticipate upfront: the Roboflow workspace endpoint wraps its project list inside a workspace object, and the classes field is a name→count map rather than a plain integer. I had to normalise both of these in the BFF before sending to the client, which is exactly the kind of thing a BFF is useful for.

4. Key Implementation Details

4.1 BFF response normalisation

Both upstream APIs have some quirks. Roboflow's workspace list endpoint wraps everything in { workspace: { projects: [...] } }, and the classes field per-project is an object like { pothole: 638, porthole: 75 }. The single-project detail endpoint nests the project under its own identifier as the key (e.g. { "new-pothole-detection": { ... } }). All of this gets flattened in the BFF so the frontend just sees a consistent { projects: [...] } shape with classes as a number.

Redmine similarly wraps collections: { issues: [...], total_count: N }, { trackers: [...] } etc. The frontend API layer extracts the array from each wrapper, so components always receive plain typed arrays and don't need to know about the envelope.

4.2 Image upload flow

Images are uploaded as multipart/form-data from the browser, buffered in Multer's memory store in the BFF, then forwarded to Roboflow's upload endpoint using a FormData object reconstructed from the buffer. The split (train / valid / test) is sent as a form field alongside the file. This avoids writing anything to disk.

4.3 Inference

The inference page supports two source modes — workspace models and public Roboflow models. I added this because my own workspace doesn't have a trained model yet (just raw annotated data), so testing with a public rock-paper-scissors model was the only way to actually demonstrate the inference flow end-to-end. The result is drawn onto a canvas element using the prediction bounding boxes, and the run is saved to localStorage so it shows up in the dashboard.

4.4 Dashboard

The dashboard pulls from four endpoints concurrently (Roboflow projects, Redmine projects, all issues for charts, open issues count). The charts use Recharts — a bar chart for priority distribution and a donut chart for status breakdown. Inference run history comes from localStorage rather than the server, which keeps it fast but means it's per-browser.

5. Trade-offs and Limitations

No authentication on the BFF

The Express server has no auth layer of its own. Anyone who can reach port 3001 can call the API. This is fine for local development but would need to be addressed before any kind of shared deployment. The straightforward fix would be to add a session/JWT middleware or put the whole thing behind a reverse proxy with basic auth.

Inference history is client-side only

Inference runs are stored in localStorage, which means they don't persist across browsers or devices and aren't queryable server-side. A small database table (even SQLite) would fix this, but I decided it wasn't necessary for the scope of this project.

Redmine running locally

The Redmine instance is a Docker container on localhost. The BFF talks to it via http://localhost:8080. If the BFF were deployed to a remote server, this would break. The Docker Compose file would need a proper service name reference instead of localhost.

No pagination on the frontend

The issues list fetches up to 50 records and the images grid fetches 24 at a time. There's no "load more" or page navigation. For the pothole dataset (9000+ images) this is a significant limitation — you're only ever seeing the first 24. Adding cursor or offset-based pagination would be the main thing I'd fix if this were production.

Roboflow upload to unversioned project

Images uploaded through the app go into the raw unversioned pool. Generating a new dataset version (with preprocessing and augmentation) still has to be done in the Roboflow web UI. The API does support version creation but the preprocessing config is complex enough that I didn't expose it in this UI.

6. Challenges

Roboflow inference endpoint. The documented inference URL format changed between API versions. The app originally built requests against api.roboflow.com/infer but the actual working endpoint is on detect.roboflow.com/{project}/{version} with the image sent as multipart/form-data. Took a bit of trial and error with cURL before I got the right shape.

Redmine first-time setup. The official Redmine Docker image ships with no trackers, issue statuses, or priorities configured — those are normally set up through the admin UI wizard on first run. I automated the seed data using a Ruby script inside the container (direct ActiveRecord calls) to avoid having to click through the setup manually every time the container restarts.

React tree crashes from API shape mismatches. A few places in the component tree were calling .map() on what they expected to be an array but was actually the envelope object (e.g. { trackers: [...] }). Because React has no error boundary by default, this wiped the whole page. I fixed the root cause in the API layer and added an ErrorBoundary so future issues show an error message instead of a blank screen.

7. What I Would Do Differently

  • Add an auth layer from the start — even basic JWT middleware rather than nothing
  • Persist inference history server-side so it's shared across sessions
  • Write tests alongside features rather than at the end
  • Use a proper DB migration tool (Prisma or Drizzle) instead of relying on the host app's schema — would make Redmine seed data reproducible without manual scripts
  • Add pagination to the image grid and issue list from day one; retrofitting it is more work than building it initially

8. Repository and Setup

Source: https://github.com/meghnadh7/otto

To run locally: 

# 1. Clone and install
git clone https://github.com/meghnadh7/otto.git
cd otto

# 2. Add credentials
cp backend/.env.example backend/.env
# Edit backend/.env: ROBOFLOW_API_KEY, ROBOFLOW_WORKSPACE, REDMINE_API_KEY

# 3. Start Redmine
docker run -d --name redmine -p 8080:3000 redmine

# 4. Start backend + frontend
cd backend && npm install && npm run dev
cd ../frontend && npm install && npm run dev

Or with Docker Compose (builds both services): 

docker compose up --build
