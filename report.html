VisionTrack – Technical Report

Meghnadh Bijnapally bijnapally.m@northeastern.edu April 2026

1. Project Overview

VisionTrack is a web dashboard I built to consolidate two tools I use for computer vision work — Roboflow for dataset and model management, and Redmine for issue tracking. The goal was a single interface where I can browse datasets, upload images, run inference against a trained model, and track related bugs or tasks, without constantly switching between browser tabs.

The app has five main areas: a live dashboard (stat cards + charts), Roboflow project/image management with an inline upload flow, an inference runner with bounding-box overlay, and Redmine project/issue management including a members tab. There's a GitHub Actions CI pipeline and Docker Compose setup for the full stack.

2. Architecture

I used a Backend-for-Frontend (BFF) pattern. The React app only ever talks to the Express server at /api/* — it never hits Roboflow or Redmine directly. The BFF handles authentication headers, response normalisation, and file forwarding. This keeps both API keys off the client entirely (they live in a .env file that's gitignored).

Figure 1 — VisionTrack system architecture. The BFF proxy keeps both API keys off the browser.

The BFF also handles a few response-shape differences between what the upstream APIs return and what a simple React component can consume directly — more on this in section 4.

On the frontend, all data fetching goes through TanStack Query v5. Each data source has a stable query key (e.g. ['roboflow-projects'], ['redmine-issues', projectId, statusFilter]), so navigating back to a page reuses the cached response instead of refetching. Query invalidation is used after mutations (image upload, issue creation) so the relevant list refreshes automatically.

4. Implementation Details

4.1 API response normalisation

Both upstream APIs use envelope wrappers that the frontend doesn't need to know about. Roboflow's workspace list returns: 

{
  "workspace": {
    "name": "Wall e",
    "projects": [
      { "id": "ws/proj", "classes": { "pothole": 638, "road": 100 }, ... }
    ]
  }
}

Two problems here: the workspace wrapper needs stripping, and classes is a name→count map rather than an integer. The frontend types it as number and renders it directly as text, so passing an object causes React to throw "Objects are not valid as a React child". I extracted the fix into a shared utility: 

// src/utils/normalise.ts
export function normaliseClasses(classes: unknown): number {
  if (typeof classes === 'number') return classes;
  if (classes && typeof classes === 'object' && !Array.isArray(classes))
    return Object.keys(classes).length;
  return 0;
}

The single-project detail endpoint has a different quirk: instead of a predictable key, it nests the project under its own identifier: { "pothole-detection-afafj": { "id": "...", "name": "..." } }. The unwrapProjectDetail utility finds the first key whose value is a non-array object (ignoring workspace) and uses that as the project root.

Redmine wraps collections too: { "trackers": [...] }, { "issue_statuses": [...] }, { "issues": [...], "total_count": N }. The frontend API layer unwraps each one before returning, so React components always get plain arrays and a consistent { data, total_count, offset, limit } shape for paginated resources.

4.2 Image upload pipeline

The upload flow goes: browser FormData → Multer memoryStorage → reconstructed FormData forwarded to Roboflow. The split assignment (train / valid / test) is sent as a separate form field. The key implementation detail is that Multer gives back a Buffer on req.file.buffer, which gets appended to a new form-data instance alongside the filename and MIME type: 

const fd = new FormData();
fd.append('file', buffer, { filename, contentType: mimetype });
fd.append('split', split);
return roboflowClient.post(uploadUrl, fd, {
  headers: { ...fd.getHeaders() },
  params: { api_key: ROBOFLOW_API_KEY, name: filename, split },
});

Nothing touches the filesystem. The buffer lives in memory for the duration of the request and is GC'd afterwards.

4.3 Inference runner

Roboflow inference goes to detect.roboflow.com/{project}/{version}, not api.roboflow.com. The endpoint accepts either a URL (passed as a query param ?image=<url>) or a raw image file as multipart/form-data. The backend supports both; the frontend defaults to URL-based for convenience.

The response looks like: 

{
  "predictions": [
    { "x": 320, "y": 240, "width": 100, "height": 80,
      "confidence": 0.91, "class": "pothole" }
  ],
  "image": { "width": 640, "height": 480 }
}

Bounding boxes are drawn onto a <canvas> element laid over the source image. Coordinates are in center-point format (x, y = centre of box), so the canvas draw call converts them: ctx.strokeRect(x - w/2, y - h/2, w, h). Each completed run is appended to a inferenceHistory array in localStorage, storing the detected class names, top confidence score, project ID, version, and timestamp. The dashboard reads this array on mount to populate the "Recent Inference Runs" section.

The inference page also has a toggle between workspace models and public Roboflow models. My own workspace only has unannotated raw data (no trained version), so I added public model support specifically to demo the feature — the rock-paper-scissors model at rock-paper-scissors-sxsw/14 returns predictions reliably without needing a trained workspace model.

4.4 Dashboard data loading

The dashboard fires five queries on mount. TanStack Query runs them in parallel: 

useQuery({ queryKey: ['roboflow-projects'],        queryFn: listRoboflowProjects })
useQuery({ queryKey: ['redmine-projects'],         queryFn: listRedmineProjects })
useQuery({ queryKey: ['redmine-issues-recent'],    queryFn: () => listIssues({ limit: 10 }) })
useQuery({ queryKey: ['redmine-issues-open'],      queryFn: () => listIssues({ status_id: 'open', limit: 1 }) })
useQuery({ queryKey: ['redmine-issues-all-chart'], queryFn: () => listIssues({ limit: 100 }) })

The priority bar chart and status donut chart are derived via useMemo from the all-issues result. Each chart entry is built by grouping issues by issue.priority.name or issue.status.name and counting occurrences. Colours are mapped from a static object keyed by name (e.g. Urgent → #f97316) so the chart colours are consistent regardless of the order Redmine returns statuses.

5. Authentication — Keycloak Integration

After the initial build I added a full OIDC authentication layer using Keycloak 24. The goal was to have a single login that also auto-provisions accounts in Redmine and Roboflow so users don't need separate credentials for each system.

5.1 How it fits together

Keycloak runs as a fourth Docker Compose service on port 8180. It imports the realm config on startup from keycloak/realm-export.json, which defines the visiontrack realm, a public PKCE client (visiontrack-app), and three roles: admin, developer, and reporter.

The frontend uses keycloak-js with onLoad: 'login-required' — if the browser has no active Keycloak session it's redirected to the login page before the React app even renders. After login succeeds, an Axios interceptor attaches Authorization: Bearer <token> to every outgoing request, and calls keycloak.updateToken(30) first so the token is always refreshed if it's within 30 seconds of expiry: 

client.interceptors.request.use(async (config) => {
  if (keycloak.authenticated) {
    await keycloak.updateToken(30).catch(() => keycloak.login())
    config.headers['Authorization'] = `Bearer ${keycloak.token}`
  }
  return config
})

5.2 Backend JWT validation

The Express BFF validates incoming JWTs in a requireAuth middleware applied to all /api/roboflow and /api/redmine routes. It fetches Keycloak's public keys from the JWKS endpoint and verifies the RS256 signature: 

// src/middleware/auth.ts
const jwks = jwksClient({
  jwksUri: `${KEYCLOAK_URL}/realms/${KEYCLOAK_REALM}/protocol/openid-connect/certs`,
  cache: true,
  cacheMaxAge: 600_000,  // 10 minutes
})

jwt.verify(token, getKey, { algorithms: ['RS256'], issuer: `...` }, (err, decoded) => {
  if (err) return res.status(401).json({ error: 'Invalid or expired token' })
  req.user = decoded
  next()
})

In development mode the middleware logs a warning but passes the request through, so the app still works locally when Keycloak isn't running.

5.3 User provisioning

After the first successful login, the frontend fires a POST /api/auth/provision request (fire-and-forget) with the token claims. The backend creates a matching Redmine user via POST /users.json — it checks for an existing login first so the call is idempotent — and sends a Roboflow workspace invite. Roles are mapped from Keycloak realm roles to Redmine role IDs (admin → Manager, developer → Developer, reporter → Reporter). The sidebar shows the logged-in user's name and email pulled from the JWT claims, with a Sign out button that calls keycloak.logout().

6. Trade-offs and Limitations

Inference history stored in localStorage

Each inference run is serialised to JSON and appended to localStorage['inferenceHistory']. This means the history is per-browser, doesn't survive a cache clear, and can't be queried server-side. The dashboard slice reading it is also computed client-side during render with no loading state. A simple server-side log table (even SQLite) would fix persistence and make the data shareable, but it was out of scope for this version.

Redmine service URL is hardcoded to localhost

REDMINE_URL=http://localhost:8080 in .env means the BFF expects Redmine on the same machine. In the Docker Compose setup this breaks if the BFF container tries to reach localhost (which would be its own loopback, not the host). The Compose file would need REDMINE_URL=http://redmine:3000 using the service name. Currently the Compose setup is included for the frontend/backend build pipeline but Redmine is expected to be started separately.

No frontend pagination

The images grid fetches 24 images and the issues list fetches up to 50. Both the Roboflow and Redmine APIs support offset-based pagination (?offset=N&limit=N), and the backend already passes these through, but the frontend doesn't expose any "load more" controls. For the pothole dataset with 9,000+ images, this means you're always looking at the same 24. It's the most obvious missing piece for a production-grade version of this.

7. Challenges Encountered

Roboflow inference URL

The Roboflow docs reference api.roboflow.com/infer in some places, but the actual working inference endpoint is detect.roboflow.com/{project}/{version}. The original frontend was sending requests to the wrong base URL, getting 404s. I diagnosed it by running a direct cURL against both hosts with a test image and confirmed the correct endpoint through trial and error. The backend now uses a separate ROBOFLOW_INFER_BASE_URL constant pointing at detect.roboflow.com.

Redmine Docker image ships with no seed data

The official redmine Docker image starts with a blank database — no issue trackers, no statuses, no priorities. Normally you'd go through the web admin wizard, but I needed it scriptable. The Rails runner command requires SECRET_KEY_BASE set as an env variable or it refuses to start. Once I had that, I could seed the data using ActiveRecord directly: 

docker exec -e SECRET_KEY_BASE=... redmine bash -c \
  "RAILS_ENV=production bundle exec rails runner '
    Tracker.create!(name: \"Bug\",     default_status: IssueStatus.find_by(name: \"New\"))
    Tracker.create!(name: \"Feature\", default_status: IssueStatus.find_by(name: \"New\"))
    ...
  '"

Blank page from API shape mismatch

Several frontend components were calling .map() on query results typed as T[], but the actual data returned from the BFF was still the envelope object (e.g. { trackers: [...] } instead of the array directly). Because the optional chaining in statuses?.map(...) only guards against null/undefined, it still tried to call .map on a plain object — which is undefined on objects — and threw TypeError: statuses.map is not a function. Without a React error boundary, this crash propagated up and unmounted the entire tree, leaving a blank page with no visible error.

The fix was two-part: normalise the unwrapping in the frontend API layer (each function now extracts the array from the envelope before returning), and add an ErrorBoundary component wrapping the router so future crashes show an error message instead of a blank screen.

8. Testing

Tests are in backend/src/__tests__/ using Jest + ts-jest + Supertest. There are three suites:

  • normalise.test.ts — 16 unit tests covering normaliseClasses, normaliseProject, unwrapProjectDetail, and normalisePaginatedIssues. These run with no external dependencies.
  • roboflow.route.test.ts — 5 integration tests for the Roboflow routes using a mock of the service layer. Covers the workspace envelope unwrapping, classes normalisation, and error forwarding.
  • redmine.route.test.ts — 6 integration tests for the Redmine routes, checking the issues envelope normalisation, query param forwarding, and error cases.

The service layer is mocked at the module level with jest.mock('../services/...'), so no real HTTP calls are made during tests. The npm test script is also wired into the GitHub Actions CI job, so tests run on every push.

Writing the tests retroactively (after building the feature) uncovered one real bug: unwrapProjectDetail was picking the first key from the response object regardless of whether its value was actually a nested project object — it would have matched a string-valued key like "id" and tried to use "ws/project-name" as a project record. The test for the fallback case caught this; the fix was to only select keys whose value is a non-array object.

9. Tools and AI Assistance

I used Gemini Pro as an AI coding assistant throughout this project for several things:

  • Boilerplate and scaffolding — initial Express route structure, TanStack Query hook patterns, and Tailwind component layout. Having it generate the skeleton let me focus on the integration-specific logic rather than writing repetitive setup code.
  • Frontend design decisions — dark theme colour palette, card layout, table vs grid choices per section. I'd describe what I needed and iterate on the output; most of the Tailwind class combinations in the components came from those back-and-forths.
  • Bug investigation — when the dashboard was showing a blank page and I couldn't immediately see why, describing the symptom ("whole page blank, no console error visible") to Gemini helped narrow it down to the missing React ErrorBoundary and the .map() on a non-array from the API response. I then traced through the actual code to confirm and fix it.
  • Minor things — regex for input validation, SVG path syntax for the bounding-box canvas draw, the correct Redmine API parameter name for filtering open issues (status_id=open rather than a numeric ID).

The core integration logic — BFF route structure, Roboflow response normalisation, the multipart upload pipeline, Redmine seed setup inside Docker — was written and debugged by hand. Gemini was most useful as a fast reference and first-draft generator rather than as a complete solution provider.

10. What I Would Do Differently

  • Add Keycloak (or any OIDC provider) from day one rather than bolting it on later — retrofitting auth meant touching the BFF entry point, every route file, the frontend provider tree, and the Compose config all at once
  • Store inference runs in a server-side table rather than localStorage; would also let the dashboard show history across devices and make runs queryable
  • Add offset-based pagination to the image grid and issue list before the first demo — retrofitting pagination into components that weren't designed for it is more work
  • Write tests for each feature as it was built rather than at the end; the unwrapProjectDetail bug above would have been caught immediately
  • Fix the Docker Compose setup so Redmine is included as a service with the correct internal URL, rather than relying on it being started separately on the host
